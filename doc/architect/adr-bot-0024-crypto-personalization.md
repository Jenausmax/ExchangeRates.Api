# ADR-BOT-0024: Персонализация криптовалют в Telegram-боте

**Статус**: Предложен
**Дата**: 2026-03-24

## Контекст

В BOT-0023 реализована базовая интеграция с KriptoService: при нажатии кнопки "Крипто" пользователь видит курсы **всех 10 монет** (BTC, ETH, SOL, XRP, BNB, USDT, DOGE, ADA, TON, AVAX). Для фиат-валют уже существует механизм персонализации (поле `UserDb.Currencies`, inline-клавиатура с toggle, callback `toggle_{CURRENCY}` / `save_currencies`). Пользователи хотят видеть только интересующие их монеты при нажатии "Крипто", аналогично тому, как они выбирают фиат-валюты.

### Ограничения

- KriptoService API уже поддерживает фильтрацию: `GET /api/crypto/latest?symbols=BTC,ETH&currencies=RUB`
- Бот использует `IKriptoApiClient.GetLatestPricesAsync()`, который сейчас **не передает** параметр `symbols`
- Модель `UserDb` хранит настройки в CSV-строках (проверенный паттерн)
- Telegram ограничивает inline-клавиатуру: максимум 8 кнопок в ряду, до 100 кнопок всего (10 монет -- укладываемся)

## Варианты решения

### Вариант 1: Новое поле `CryptoCoins` в UserDb (CSV, nullable) -- ВЫБРАН

Добавить поле `CryptoCoins` (string, nullable) в `UserDb` и `CurrentUser`. `NULL` = показывать дефолтный набор. CSV-строка = пользовательский выбор.

**Плюсы:**
- Полная аналогия с существующим `Currencies` -- единообразие паттерна
- Минимальные изменения в коде: новое поле + методы в `IUserService` + callback-обработчики
- EF миграция добавляет nullable-колонку -- безопасно для существующих данных
- Фильтрация на стороне KriptoService API (передаем `symbols`) -- меньше трафика

**Минусы:**
- Ещё одно CSV-поле в UserDb (но это уже устоявшийся паттерн в проекте)
- EF миграция (но тривиальная, ALTER TABLE ADD COLUMN)

### Вариант 2: Отдельная таблица `UserCryptoDb` (many-to-many)

Создать таблицу `UserCryptoDb(UserId, Symbol)` для хранения выбора.

**Плюсы:**
- Нормализованная структура БД
- Возможность добавить метаданные (порядок отображения, дата добавления)

**Минусы:**
- Нарушает устоявшийся паттерн проекта (все персонализации хранятся CSV в UserDb)
- Усложнение: JOIN-запросы, новый репозиторий, новая модель
- Для 10 монет нормализация -- overengineering (YAGNI)
- Потребуется рефактор существующего `GetCollection`-подхода

### Вариант 3: Фильтрация на стороне бота (без хранения)

Получать все 10 монет от KriptoService и фильтровать в `FormatCryptoPrices()`.

**Плюсы:**
- Нет миграции, нет изменений в UserDb

**Минусы:**
- Избыточный трафик (всегда запрашиваем все 10)
- Всё равно нужно хранить выбор пользователя где-то
- Не использует существующую фильтрацию KriptoService API

## Решение

**Выбран Вариант 1** -- новое CSV-поле `CryptoCoins` в `UserDb`.

### Обоснование

1. **Единообразие** -- точно такой же паттерн используется для `Currencies` (фиат) и `NewsTimes` (расписание). Разработчик, знакомый с одним, сразу поймёт другой.
2. **Простота** -- одна миграция, одно поле, минимальные изменения в интерфейсах.
3. **Эффективность** -- передаём `symbols` в KriptoService API, получаем только нужные данные.
4. **YAGNI** -- для 10 монет нормализация не нужна. Если список вырастет до 50+, можно будет мигрировать (Strangler Fig).

### Дефолтное поведение при NULL

`NULL` = показывать **все 10 монет** (текущее поведение, обратная совместимость). Это отличается от фиат-валют (NULL = 5 из 10), и намеренно: пользователь, не настроивший крипто, ожидает видеть то же, что видел до появления фичи.

### Альтернатива для дефолта: подмножество (BTC, ETH, SOL, XRP, BNB)

Рассмотрена и отклонена. Причина: нарушит ожидания существующих пользователей, которые привыкли видеть все 10 монет.

## Последствия

### Положительные
- Пользователь получает персонализированный список монет
- Уменьшается объём данных от KriptoService (при фильтрации)
- Код следует устоявшемуся паттерну -- легко поддерживать
- UX аналогичен выбору фиат-валют -- пользователь уже знаком с механикой

### Отрицательные
- Новая EF-миграция (минимальный риск: nullable колонка)
- Новые callback-данные увеличивают switch/if-цепочку в `CallbackMessageCommand`
- `ConcurrentDictionary` для pending-состояния -- ещё один static dict в `CommandService`

### Риски
- **Конфликт callback-префиксов**: `toggle_` уже используется для фиат-валют. Решение: использовать `toggle_crypto_` для криптомонет.
- **Разрастание CommandService**: класс уже 810 строк. Рекомендация на будущее (вне scope BOT-0024): вынести крипто-логику в отдельный handler.
- **Потеря in-memory состояния**: `_pendingCryptoSelections` теряется при рестарте бота. Это допустимо -- при рестарте пользователь просто начнёт выбор заново (аналогично текущему поведению для `_pendingSelections`).
